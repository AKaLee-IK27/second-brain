# Requirement: Interactive Outline Upgrade for Detail Screens

**Date:** 2026-04-14
**Status:** Reviewed — Scope Expanded
**Author:** Scribe

## User Story

As a reader of knowledge content (topics, sessions, agents, skills), I want an interactive, collapsible outline panel in the detail screen's right metadata rail so that I can quickly navigate to any section of long documents, see my current reading position highlighted in real-time, and hide the outline when I don't need it.

## Context

### Current State

Four detail pages render markdown content with a right metadata rail:

| Page | File | Has Outline | Clickable | Has Collapse | Has Active Tracking |
|------|------|-------------|-----------|--------------|---------------------|
| Topic Detail | `TopicDetailPage.tsx` | Yes (static) | No (no handler) | No | No |
| Session Detail | `SessionDetailPage.tsx` | Yes (static) | No (no handler) | No | No |
| Agent Detail | `AgentDetailPage.tsx` | No | N/A | No | No |
| Skill Detail | `SkillDetailPage.tsx` | No | N/A | No | No |

The outline is extracted via regex (`/^(#{1,6})\s+(.+)$/gm`) from the markdown body and rendered as a static `<ul>` list in the right rail. Items have `cursor-pointer` styling but no `onClick` handler — they are visually clickable but do nothing.

There is a separate `OutlinePanel.tsx` component in `src/components/layout/` that targets the TipTap editor (`.ProseMirror`), which is a different context and not affected by this requirement.

### Target State

All four detail pages will share a consistent, interactive outline component with:
- **Click-to-scroll**: Clicking a heading scrolls the article body to that heading with smooth animation.
- **Collapse/expand**: A toggle button in the outline header collapses the list to save vertical space, with a smooth height animation.
- **Active heading tracking**: As the user scrolls the article, the outline highlights the heading currently in view.
- **Slug-based heading IDs**: Headings in the rendered markdown receive generated `id` attributes for reliable scroll targeting.
- **State persistence**: The collapsed/expanded state persists across page navigations within the same browser session via `sessionStorage`.
- **Full heading depth**: H1-H6 headings are extracted and displayed in the outline with appropriate indentation.
- **Keyboard navigation**: Arrow keys navigate between outline items, Enter/Space activates a heading, Escape collapses the outline.

## Acceptance Criteria

### A. Shared Outline Component

- [ ] Given a new `ArticleOutline` component is created in `src/components/shared/ArticleOutline.tsx`, when it is rendered with a `headings` array, then the heading list displays one item per heading with the correct text, indentation level, and the header row shows the title and toggle icon
- [ ] Given the `ArticleOutline` component, when `isCollapsed` is true, then only the header row is visible (title + toggle icon) and the heading list is hidden
- [ ] Given the `ArticleOutline` component, when `isCollapsed` is false, then the heading list is rendered below the header with each item indented according to its `level` (H1 no indent, H2 indented, H3 more indented, H4/H5/H6 progressively more indented at 16px per level — one Tailwind spacing unit)
- [ ] Given the `ArticleOutline` component, when a heading item is clicked, then it calls an `onHeadingClick` callback with the heading's `id`
- [ ] Given the `ArticleOutline` component, when `activeHeadingId` matches a heading item's `id`, then that item is visually highlighted with `text-primary` color and a `primary-container` left indicator bar

### B. Click-to-Scroll Navigation

- [ ] Given a detail page with rendered markdown headings, when the user clicks an outline item, then the browser scrolls to the corresponding heading element with `behavior: 'smooth'` and `block: 'start'`
- [ ] Given the scroll target heading, when the scroll completes, then the heading's top edge aligns with the scroll container's top edge plus the value of `scroll-padding-top`, with a tolerance of ±4px
- [ ] Given a heading with special characters in its text (e.g., "What's New?", "C++ Basics"), when the scroll target is resolved, then the scroll succeeds because the heading `id` uses a URL-safe slug (lowercase alphanumeric and hyphens only)

### C. Heading ID Generation

- [ ] Given a markdown body string, when headings are extracted, then each heading is assigned a unique `id` generated by: converting text to lowercase, replacing spaces with hyphens, removing non-alphanumeric characters except hyphens, and appending a numeric suffix if duplicates exist (e.g., `introduction`, `introduction-1`, `introduction-2`)
- [ ] Given the markdown rendering pipeline, when the article body is rendered, then each heading element (`h1`, `h2`, `h3`, `h4`, `h5`, `h6`) in the DOM has an `id` attribute matching the generated slug
- [ ] Given a heading text that, after slug generation, produces an empty string (e.g., text containing only special characters like `## !!!`), when the ID is generated, then a fallback ID is created using the heading level and index (e.g., `heading-h1-0`)

### D. Active Heading Tracking (Scroll Spy)

- [ ] Given the article body is scrollable, when the user scrolls, then the outline highlights the heading whose top edge has the smallest non-negative offset from the scroll container's top edge
- [ ] Given multiple headings are visible in the viewport, when determining the active heading, then the topmost visible heading is selected as active
- [ ] Given the user scrolls to the very top of the article, when no heading is above the viewport midpoint, then no outline item is highlighted (activeHeadingId is null)
- [ ] Given the scroll event fires rapidly, when tracking active headings, then the tracking mechanism (IntersectionObserver or scroll listener) does not trigger more than 10 state updates per second (100ms minimum interval) to avoid performance degradation

### E. Collapse/Expand Toggle

- [ ] Given the outline section header, when the user clicks the toggle icon (the existing `segment` MaterialIcon in the header), then the outline list toggles between collapsed and expanded states
- [ ] Given the outline is collapsed, when it renders, then the toggle icon visually indicates the collapsed state (e.g., icon rotation or icon change to `expand_more`)
- [ ] Given the outline is expanded, when it renders, then the toggle icon visually indicates the expanded state (e.g., icon rotation or icon is `segment`)
- [ ] Given the outline collapses or expands, when the state changes, then the transition animates with a smooth height change over 200ms using `ease-in-out` timing
- [ ] Given the outline collapsed state, when the component unmounts and remounts, then the state is restored from `sessionStorage` under the key `outline-collapsed-state`
- [ ] Given no persisted state exists in `sessionStorage`, when the component mounts, then the outline defaults to expanded

### F. Consistent Application Across All Detail Pages

- [ ] Given the TopicDetailPage, when it renders, then it uses the shared `ArticleOutline` component with all interactive features (click-to-scroll, collapse, active tracking)
- [ ] Given the SessionDetailPage, when it renders, then it uses the shared `ArticleOutline` component with all interactive features
- [ ] Given the AgentDetailPage, when it renders and the markdown body contains headings, then it displays the outline section using the shared `ArticleOutline` component
- [ ] Given the SkillDetailPage, when it renders and the markdown body contains headings, then it displays the outline section using the shared `ArticleOutline` component
- [ ] Given a detail page where the markdown body contains zero headings, when it renders, then no outline section is displayed in the right rail

### G. Visual Design Compliance

- [ ] Given the outline panel styling, when rendered, then it follows the Monolithic Lexicon design system: `font-mono` text-[10px] uppercase tracking-[0.2em] for the header, `font-headline` text-sm for heading items, `text-on-surface-variant` for inactive items, `text-primary` for active item
- [ ] Given an outline heading item, when hovered, then it transitions to `text-primary` with a 150ms duration
- [ ] Given an outline heading item at level 2 or 3, when rendered, then it has a left padding offset (`pl-4` for H2, `pl-8` for H3) consistent with the existing design

### H. H4-H6 Heading Support

- [ ] Given a markdown body with H4, H5, or H6 headings, when the outline is extracted, then all headings from H1 through H6 appear in the outline list
- [ ] Given an H4, H5, or H6 heading item in the outline, when rendered, then it has a left padding offset of `pl-12` for H4, `pl-16` for H5, and `pl-20` for H6
- [ ] Given an H4, H5, or H6 heading item, when clicked, then it scrolls to the corresponding heading with the same behavior as H1-H3 items
- [ ] Given an H4, H5, or H6 heading is active, when tracked by the scroll spy, then it is highlighted in the outline with the same visual indicator as H1-H3 items

### I. Keyboard Navigation

- [ ] Given the outline expands, when the transition completes, then focus is automatically set to the first heading item in the list
- [ ] Given the outline is expanded and focused, when the user presses ArrowDown, then focus moves to the next heading item in the list (wrapping to the first item when at the end)
- [ ] Given the outline is expanded and focused, when the user presses ArrowUp, then focus moves to the previous heading item in the list (wrapping to the last item when at the first)
- [ ] Given a heading item is focused, when the user presses Enter or Space, then the `onHeadingClick` callback is triggered and the browser scrolls to that heading
- [ ] Given the outline is expanded and focused, when the user presses Escape, then the outline collapses and focus returns to the toggle button
- [ ] Given the outline is collapsed and the toggle button is focused, when the user presses Enter or Space, then the outline expands
- [ ] Given keyboard navigation is active, when a heading item receives focus, then it is visually indicated with a focus ring (`outline-2 outline-primary`)

### J. State Persistence

- [ ] Given the user collapses the outline, when the state is saved, then `sessionStorage` stores the key `outline-collapsed-state` with value `"true"`
- [ ] Given the user expands the outline, when the state is saved, then `sessionStorage` stores the key `outline-collapsed-state` with value `"false"`
- [ ] Given the user navigates to a different detail page, when the `ArticleOutline` component mounts, then it reads `sessionStorage` for `outline-collapsed-state` and applies the persisted value (the collapsed state is shared across all detail pages within the same browser tab)
- [ ] Given the browser tab is closed and reopened, when the `ArticleOutline` component mounts, then the outline defaults to expanded (sessionStorage is cleared on tab close)

## Constraints

### Technical
- Must use React 19 + TypeScript + Tailwind CSS v4 (no framework changes)
- Must use the heading extraction regex pattern (`/^(#{1,6})\s+(.+)$/gm`) — H1-H6 are included
- The `MarkdownRenderer` component (`src/components/shared/MarkdownRenderer.tsx`) must be modified or wrapped to inject `id` attributes into heading elements, OR heading IDs must be injected post-render via DOM manipulation
- Must use `IntersectionObserver` or scroll event listeners for active heading tracking — no external libraries
- The scroll container is the center panel `div` with `overflow-y-auto`, not the window
- Must not break the existing `OutlinePanel.tsx` in `src/components/layout/` which is used for the TipTap editor context (the new component is named `ArticleOutline` and placed in `src/components/shared/` to avoid naming conflict)
- State persistence must use `sessionStorage` (not `localStorage`) — state clears when the browser tab is closed

### Design
- Must follow the Monolithic Lexicon design system as defined in `docs/ai/requirements/monolithic-lexicon-ui-overhaul.md`
- Collapse toggle must use the existing header layout pattern (header text on left, icon on right)
- Active heading indicator must use `primary-container` color (`#58a6ff`) for the left accent bar

### Performance
- Must use `IntersectionObserver` for active heading tracking to avoid per-scroll-event re-renders
- Heading ID generation for documents up to 50,000 characters must complete in under 50ms

## Edge Cases

### No Headings
- When the markdown body contains no H1-H6 headings, the outline section is not rendered at all (existing behavior preserved).

### Duplicate Heading Texts
- When multiple headings have identical text (e.g., two "Summary" sections), each receives a unique slug ID with numeric suffix (`summary`, `summary-1`).

### Very Long Heading Texts
- When a heading text exceeds the outline panel width, it is truncated with ellipsis (`truncate` / `overflow-hidden text-ellipsis whitespace-nowrap`), with the full text visible on hover via the browser's native `title` attribute.

### Deeply Nested Headings
- H1 through H6 are all supported (matching the regex `/^(#{1,6})\s+(.+)$/gm`). Headings beyond H6 are ignored and do not appear in the outline.

### MarkdownRenderer Injects Different Heading Structure
- If the `MarkdownRenderer` wraps headings in additional elements (e.g., `<div><h2>...</h2></div>`), the `id` attribute must be on the outermost wrapper element, and scroll targeting must resolve to that wrapper element's bounding rectangle.

### Rapid Scrolling
- When the user scrolls rapidly through the document, the active heading updates at most once per 100ms and does not cycle through more than 2 distinct heading IDs within any 500ms window.

### Component Unmount During Scroll
- When the user navigates away from a detail page while scroll tracking is active, all scroll listeners and observers are cleaned up to prevent memory leaks.

## Out of Scope

- Persisting collapsed/expanded state across browser tab close/reopen (localStorage)
- Drag-to-reorder or editable outline items
- Outline search/filter within the heading list
- Changes to the `MarkdownRenderer` component's core markdown parsing logic (only heading ID injection is in scope)
- Changes to the editor-focused `OutlinePanel.tsx` in `src/components/layout/`
- Mobile-responsive adaptations for the outline panel
- Exporting or sharing the outline as a separate document

## Implementation Notes

### Recommended Approach for Heading ID Injection

Two viable approaches exist:

1. **Modify `MarkdownRenderer`**: Pass heading IDs through the markdown rendering pipeline (e.g., using `marked`'s renderer options to add `id` attributes to heading tokens). This is the cleanest approach.

2. **Post-render DOM injection**: After `MarkdownRenderer` renders, use a `useEffect` to query all heading elements in the article and assign IDs based on the extracted headings array. This is simpler but less elegant.

The `marked` library (already a dependency) supports custom renderers that can add `id` attributes to headings. This is the recommended approach.

### Recommended Approach for Active Heading Tracking

Use `IntersectionObserver` with a root margin that targets the top portion of the scroll container. Each heading element gets observed, and the observer callback updates the active heading state. This is more performant than scroll event listeners and avoids manual position calculations.

### Component Naming

The existing `src/components/layout/OutlinePanel.tsx` targets the TipTap editor. The new component for detail screen outlines should be named `ArticleOutline` or placed in a different directory (e.g., `src/components/shared/ArticleOutline.tsx`) to avoid naming conflicts.
